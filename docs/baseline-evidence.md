# `db-v1.0.0` baseline evidence

This document contains non-sensitive evidence for the canonical baseline
candidate. Production credentials, row data, backup archives, ledger exports,
and complete database dumps remain outside Git.

- Schema freeze began at `2026-07-17T14:37:33Z`, recorded on
  [`sal-database#3`](https://github.com/diese-tech/sal-database/issues/3#issuecomment-5004379856).
- Ordinary row-level league operations continue during the freeze.
- Source database: PostgreSQL 17.
- Source repository commit: `372dcf613cd219e74fbdcacc3226fc160f692eb3`.
- Active baseline version: `20260717143900`.

The live migration ledger contained 14 captured historical versions.
Archived site migrations `019` through `025` were absent; their effects were
not all present in production. Adoption must use the captured ledger as an
allowlist and must not execute or repair those absent versions.

The migration file differs from the `public` dump because it also restores the
application-owned `match-screenshots` Storage bucket and its two verified
policies. Contract tests independently assert all 21 public RLS policies, the
service-role grants, the two Realtime publication members, and Storage state.
The SMITE fixture is generated by `diese-tech/smite-content-sync` and loaded as
a separate deterministic seed input; it is not production league data.

## Preserved state and deferred hardening

The baseline deliberately preserves production state rather than silently
folding unshipped hardening into v1. In particular, later work must use
reviewed forward migrations for function `search_path` pinning, match-season
constraints, pending-result uniqueness, timeout handling, and shared-schema
alignment.

Production was read only throughout capture and verification. The production
ledger is unchanged, the baseline DDL has not run there, and no release tag has
been created. Those steps remain protected by the adoption runbook and a
commit-bound recovery attestation.
